**1b. Write a simple program to add two integers and display the result using C++ (5½ Mks)**

```cpp
#include <iostream>

int main() {
    int num1, num2, sum;

    // Prompt the user to enter two integers
    std::cout << "Enter two integers: ";
    std::cin >> num1 >> num2;

    // Calculate the sum
    sum = num1 + num2;

    // Display the result
    std::cout << "Sum of " << num1 << " and " << num2 << " is: " << sum << std::endl;

    return 0;
}
```

**Explanation:**

1. `**#include <iostream>`**:  Includes the iostream library, which provides input/output functionalities like `std::cout` (for output) and `std::cin` (for input).
2. `**int main() { ... }`**:  The `main` function is the entry point of every C++ program.
3. `**int num1, num2, sum;**`:  Declares three integer variables: `num1`, `num2` to store the two input numbers, and `sum` to store their sum.
4. `**std::cout << "Enter two integers: ";**`:  Displays a message to the user prompting them to enter two integers.
5. `**std::cin >> num1 >> num2;**`:  Reads two integers from the user's input and stores them in the variables `num1` and `num2` respectively.
6. `**sum = num1 + num2;**`:  Calculates the sum of `num1` and `num2` and assigns the result to the `sum` variable.
7. `**std::cout << "Sum of " << num1 << " and " << num2 << " is: " << sum << std::endl;**`:  Displays the result. It prints the message "Sum of ", followed by the values of `num1`, " and ", `num2`, " is: ", and finally the value of `sum`. `std::endl` inserts a newline character at the end of the output, moving the cursor to the next line.
8. `**return 0;**`:  Indicates that the program executed successfully.

**2a. Write short notes on the following criteria for language evaluation and comparison (12 Mks)**

**(i) Expressivity:** Expressivity refers to the power and ease with which a programming language can express a wide range of computations and programming concepts. A more expressive language allows programmers to write code that is concise, clear, and reflects the intended logic directly.

**(ii) Well-Definedness:** Well-definedness refers to the clarity, precision, and completeness of the language's specification and semantics. A well-defined language has clear rules for syntax, semantics, and behavior, leaving little room for ambiguity or undefined behavior.

**(iii) Portability:** Portability refers to the ability of programs written in a language to be executed on different computing platforms (operating systems, hardware architectures) with minimal or no modification.

**(iv) Efficiency:** Efficiency relates to how well a program utilizes computing resources (CPU time, memory, energy) during execution. It can refer to both execution speed (runtime efficiency) and memory usage (space efficiency).

**(v) Pedagogy:** Pedagogy refers to how suitable a language is for teaching and learning programming concepts. A pedagogically sound language is easy to learn, has a clear structure, and effectively illustrates fundamental programming principles.

**(vi) Generality:** Generality refers to the breadth of application domains for which a language is well-suited. A general-purpose language can be used for a wide variety of tasks, from system programming to web development to data science.

**(i) Itemize the four general methods of implementing a programming language? (2½ Mks)**

The four general methods of implementing a programming language are:

1. **Compilation:**  Translating the entire source code into machine code or another lower-level form before execution.
2. **Interpretation:**  Executing the source code line by line, directly, without prior translation to machine code.
3. **Hybrid Approach (Compilation to Bytecode + Interpretation/JIT Compilation):** Compiling source code to an intermediate bytecode and then using a virtual machine to interpret or Just-In-Time (JIT) compile the bytecode to machine code during runtime.
4. **Translation (Source-to-Source Compilation):** Translating source code from one programming language to another high-level programming language, which is then further processed (e.g., compiled or interpreted). (Less common as a primary implementation method for general-purpose languages but relevant for certain domain-specific languages or language extensions).

**(ii) What are the application domains of programming languages? (12 Mks)**

Programming languages are broadly grouped by the kinds of problems they were designed to solve. The major application domains are:

1. **Scientific Applications:** The first domain for which computers were widely used. These programs involve large numbers of **floating-point arithmetic computations** and make heavy use of **arrays/matrices** and **counting (for) loops**. Data structures are simple, but there is a high demand for computational **efficiency (speed)**.
  - *Example languages:* Fortran, ALGOL 60.
2. **Business Applications:** Used to produce **reports**, process **decimal numbers and character data**, and perform accurate decimal arithmetic. Emphasis is on elegant report formatting and precise handling of money/decimal values.
  - *Example languages:* COBOL (and later spreadsheet/RPG-style tools).
3. **Artificial Intelligence (AI):** Characterised by **symbolic** rather than numeric computation — manipulating symbols and **linked lists** instead of numbers. Requires great flexibility, and programs are often built to modify or generate other code at run time.
  - *Example languages:* LISP, Scheme, Prolog.
4. **Systems Programming:** Building the software that runs the machine itself — **operating systems, device drivers, and systems-support utilities**. Such software runs continuously, so it must be **fast/efficient** and provide **low-level features** to interface directly with the hardware.
  - *Example languages:* C, C++ (e.g., the UNIX operating system is written mostly in C).
5. **Web Software:** Supports content and computation on the World Wide Web. It is served by an **eclectic collection of languages**, ranging from **markup languages** (HTML/XML), through **scripting languages** (JavaScript, PHP, Perl), to **general-purpose languages** (Java).
  - *Example languages:* HTML/XML, JavaScript, PHP, Java.

*(Other specialised domains include publishing, education, and database/query processing, but the five above are the classic application domains of programming languages.)*

**3b. Itemize compilation process phases in a typical compiler (3½ Mks)**

The compilation process in a typical compiler is generally divided into several phases. These phases transform the source code step-by-step into executable code. The typical phases are:

1. **Lexical Analysis (Scanning):**
  - **Input:** Source code (text).
  - **Process:** Reads the source code character by character and groups characters into meaningful units called **tokens**. Tokens represent keywords, identifiers, operators, literals, etc.  Removes whitespace and comments.
  - **Output:** Stream of tokens.
  - **Tool:** Lexer or Scanner.
2. **Syntax Analysis (Parsing):**
  - **Input:** Stream of tokens from lexical analysis.
  - **Process:** Analyzes the structure of the token stream to determine if it conforms to the grammar (syntax rules) of the programming language. Builds a **parse tree** or **Abstract Syntax Tree (AST)** representing the grammatical structure of the program. Detects syntax errors.
  - **Output:** Parse tree or AST.
  - **Tool:** Parser.
3. **Semantic Analysis:**
  - **Input:** Parse tree or AST.
  - **Process:** Checks the meaning and consistency of the program beyond syntax. Performs type checking, scope resolution, semantic error detection (e.g., type mismatches, undeclared variables), and verifies semantic rules of the language. Annotates the AST with semantic information.
  - **Output:** Annotated AST or intermediate representation.
  - **Tasks:** Type checking, scope resolution, symbol table management, semantic error detection.
4. **Intermediate Code Generation:**
  - **Input:** Annotated AST or intermediate representation.
  - **Process:** Translates the program into an intermediate representation (IR) that is more abstract than machine code but more concrete than source code. Common IRs include three-address code, bytecode, or platform-independent assembly. IR is often designed to be platform-neutral and facilitate optimization.
  - **Output:** Intermediate code (IR).
5. **Optimization (Optional but crucial for performance):**
  - **Input:** Intermediate code (IR).
  - **Process:** Analyzes and transforms the intermediate code to improve its performance (e.g., execution speed, memory usage, code size). Optimizations can be local (within a basic block), global (across functions), or machine-dependent. Examples: constant folding, dead code elimination, loop optimization, register allocation.
  - **Output:** Optimized intermediate code (IR).
6. **Code Generation:**
  - **Input:** Optimized intermediate code (IR).
  - **Process:** Translates the optimized intermediate code into target machine code (assembly language or directly executable machine code) specific to the target processor architecture and operating system. Handles register allocation, instruction selection, and machine-specific code generation details.
  - **Output:** Target machine code (executable code).
7. **Symbol Table Management (Throughout all phases):**
  - **Process:** A symbol table is a data structure used by the compiler to store information about identifiers (variables, functions, classes, etc.) encountered in the source code. Information includes name, type, scope, memory location, etc. The symbol table is built up during lexical and syntax analysis and used in semantic analysis, optimization, and code generation.
8. **Error Handling (Throughout all phases):**
  - **Process:**  Error detection and reporting occur in various phases (lexical, syntax, semantic). When an error is detected, the compiler should report it to the user with informative messages, indicating the type of error and its location in the source code. Error recovery strategies may also be employed to continue compilation to find more errors.

These phases are often sequential, but there can be overlaps and iterations, especially in more complex compilers. The specific phases and their details can vary depending on the compiler design and the programming language being compiled.

**(i) Describe the term programming language (2 Mks)**

A **programming language** is a formal, artificial language designed to communicate instructions to a computer. It provides a set of vocabulary (keywords), syntax rules (grammar), and semantics (meaning) that allow programmers to write instructions (code) that a computer can understand and execute to perform specific tasks or solve problems. Programming languages bridge the gap between human thought and computer execution, enabling the creation of software applications, systems, and tools.

The three main types of grammar used in programming languages are:

1. **Regular Grammar (Type-3 Grammar):** Primarily used for lexical analysis (tokenizing). Simple rules, recognized by finite automata.
2. **Context-Free Grammar (CFG) (Type-2 Grammar):** Used for defining the syntax of programming languages (parsing). More powerful than regular grammars, recognized by pushdown automata.
3. **Context-Sensitive Grammar (CSG) (Type-1 Grammar):** More powerful than CFGs, but complex. Less commonly used for core syntax definition in compilers.

**(iii) State three advantages of Dynamic Type checking (3 Mks)**

Advantages of Dynamic Type Checking:

1. **Flexibility and Rapid Development:** Dynamic typing is more flexible and can lead to faster initial development. Programmers don't need to declare types explicitly, reducing code verbosity and allowing for quicker prototyping and scripting.
2. **Code Conciseness:** Code in dynamically-typed languages tends to be shorter and more concise because type declarations are often omitted. This can make code easier to write and sometimes faster to read for simple scripts.
3. **Polymorphism and Generics (Easier to Implement):** Dynamic typing naturally supports polymorphism and generic programming. Functions and methods can work with objects of different types without explicit type specifications or complex type parameterization. This can simplify code and increase reusability in certain situations.

**Brief Description of Type Checking:**

Type checking is the process of verifying and enforcing constraints on the types of values used in a program. It ensures that operations are performed on compatible data types, preventing type errors that can lead to unexpected behavior or crashes. Type checking can be static (at compile time) or dynamic (at runtime). It uses a type system with rules to determine type correctness and reports errors if type rules are violated.

**(ii) What are the two main uses of Type Checking? (3 Mks)**

The two main uses of Type Checking in programming languages are:

1. **Error Detection and Prevention:**
  - **Purpose:** To detect and prevent type-related errors in programs. Type errors are common sources of bugs and can lead to incorrect program behavior, crashes, or security vulnerabilities.
  - **Benefit:** Type checking helps catch these errors early in the development cycle (especially static type checking at compile time), making programs more robust and reliable. It reduces the likelihood of runtime type errors.
2. **Code Optimization and Performance:**
  - **Purpose:** Type information obtained through type checking can be used by compilers to perform optimizations and generate more efficient code.
  - **Benefit:** In statically-typed languages, knowing the types of variables and expressions at compile time allows compilers to make assumptions and perform optimizations such as:
    - **Memory allocation optimization:** Allocating appropriate memory based on type size.
    - **Instruction selection:** Choosing more efficient machine instructions based on operand types.
    - **Inlining and other optimizations:**  Optimizations that are safer and more effective when type information is available.
  - **Result:** Type checking can contribute to improved program performance, especially in compiled languages.

**(ii) The syntax of a programming language can be interpreted using various formal and informal techniques. Briefly describe these techniques. (4½ Mks)**

The syntax of a programming language, which defines the valid structure of programs, can be described and interpreted using both formal and informal techniques:

- **Formal Techniques:** These techniques use mathematical and rigorous methods to define syntax precisely and unambiguously.
  1. **Formal Grammars (Context-Free Grammars - CFGs):**
    - **Description:** Use production rules to define the grammatical structure of a language. CFGs are commonly used to specify the syntax of programming languages. Examples include Backus-Naur Form (BNF) and Extended BNF (EBNF) notations.
    - **Interpretation:** Parsers (e.g., LL parsers, LR parsers) are automatically generated or constructed based on formal grammars. These parsers take source code as input and check if it conforms to the grammar, producing a parse tree or AST if valid, or syntax error messages if invalid.
    - **Example:** Defining the syntax for an `if` statement or an expression using BNF rules.
  2. **Regular Expressions:**
    - **Description:** Formal notation for describing patterns of characters. Used to define the lexical syntax of a language, i.e., the structure of tokens (keywords, identifiers, operators, literals).
    - **Interpretation:** Lexical analyzers (scanners) are built using regular expressions. They read the source code and break it into tokens based on these regular expression patterns.
    - **Example:** Defining the pattern for identifiers (e.g., starting with a letter, followed by letters or digits) or for integer literals using regular expressions.
- **Informal Techniques:** These techniques use natural language descriptions and examples to explain syntax. They are often used for documentation and teaching but are less precise than formal methods.
  1. **Natural Language Descriptions (Language Manuals/Specifications):**
    - **Description:** Syntax is described using English or other natural languages. Language manuals and specifications often use natural language to explain the rules, keywords, operators, and statement structures of a programming language.
    - **Interpretation:** Programmers learn the syntax by reading these descriptions and examples. They rely on their understanding of natural language and examples to write syntactically correct code.
    - **Example:** "In Python, an `if` statement starts with the keyword `if`, followed by a condition, a colon `:`, and an indented block of code."
  2. **Syntax Diagrams (Railroad Diagrams):**
    - **Description:** Graphical representations of syntax rules. Syntax diagrams visually illustrate the structure of language constructs using boxes, lines, and symbols.
    - **Interpretation:** Programmers can follow the diagrams to understand the valid sequences of tokens and construct syntactically correct code. Syntax diagrams are easier to understand visually than formal grammar rules for some people.
    - **Example:** Diagrammatically showing the structure of a `for` loop or a function definition, indicating required and optional parts.
  3. **Examples:**
    - **Description:** Providing numerous examples of syntactically correct code snippets. Examples illustrate how to use different language constructs and syntax rules in practice.
    - **Interpretation:** Learners infer syntax rules from examples. Examples help in understanding the practical application of syntax rules and can be more intuitive than abstract formal rules.
    - **Example:** Showing code examples of variable declarations, loops, function calls, and class definitions to illustrate the correct syntax.

**Comparison:**

- **Formal Techniques (Grammars, Regular Expressions):**
  - **Pros:** Precise, unambiguous, machine-processable, basis for automated parser/lexer generation, essential for compiler/interpreter implementation.
  - **Cons:** Can be harder for beginners to understand, require formal notation knowledge.
- **Informal Techniques (Natural Language, Diagrams, Examples):**
  - **Pros:** Easier for humans to understand, more intuitive for learning, good for documentation and teaching.
  - **Cons:** Less precise, can be ambiguous, not directly machine-processable, not sufficient for compiler/interpreter implementation alone.

In practice, programming language specifications and documentation often use a combination of both formal (for precise syntax definition) and informal techniques (for explanation and examples) to describe and interpret the syntax of a programming language effectively.

The two types of declaration are:

1. **Explicit Declaration:** Programmer explicitly specifies the type of a variable.
2. **Implicit Declaration:** Type is inferred from context or usage.

**(ii) Outline the purpose of Declarations in programming (4 Mks)**

[**Answer is the same as 1b(ii) from set 1. Refer to the answer provided for 1b(ii) in set 1 for the purposes of declarations.**]

The purposes of declarations in programming include:

1. **Type Association:** Associating a data type with a variable.
2. **Naming and Identification:** Introducing names (identifiers) for entities.
3. **Scope and Lifetime Definition:** Specifying where and how long variables exist.
4. **Initialization (Optional):** Setting initial values.
5. **Interface Definition:** Defining interfaces for functions, classes, etc.
